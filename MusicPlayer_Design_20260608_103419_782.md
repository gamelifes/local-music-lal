# MusicPlayer - 本地音乐播放器设计文档 v3

## 一、需求概述

一款纯本地使用的音乐播放器，支持 PWA 浏览器运行 + Capacitor 打包为 Android APK。不依赖网络，不上传任何数据。核心功能：本地扫描、逻辑删除、音效增强、音质识别、自定义均衡器、逐字歌词显示、桌面歌词浮窗。

#### 音频播放库选型

**选型结论**：**Howler.js** — 负责音频解码与播放控制；均衡器和音效处理仍走原生 Web Audio API `BiquadFilterNode` 管线。

**对比表**：

| 维度 | **Howler.js** | 原生 HTMLAudio + Web Audio API | Tone.js | Wavesurfer.js |
|------|-------------|------------------------------|---------|---------------|
| 包体积 | ~45KB gzipped | 0KB（浏览器内置） | ~180KB gzipped | ~120KB gzipped |
| Web Audio 集成 | 自动降级（Web Audio → HTML5 Audio），支持直接获取 AudioContext | 完全手动集成 | 原生基于 Web Audio，内置合成器/效果器 | 基于 Web Audio，侧重可视化 |
| 音频格式支持 | MP3 / FLAC / WAV / OGG / AAC / M4A / OPUS | 取决于浏览器，APE 不支持 | 同浏览器 | 同浏览器 |
| 播放控制 API | 完善：`play/pause/seek/rate/volume/fade/loop` | 手动实现 | 侧重合成而非播放控制 | 播放 + 波形可视化 |
| 移动端兼容性 | **优秀**：自动处理移动端音频限制（需用户手势后才可播放） | 需手动处理手势解锁 | 一般 | 一般 |
| 维护活跃度 | 稳定（2.x 已维护多年） | N/A | 活跃（~13K Stars） | 活跃（~8K Stars） |
| 学习成本 | **低**：类 HTMLAudioElement API | 中：需理解 AudioContext 生命周期 | 高：侧重音乐创作 | 中 |
| 许可协议 | MIT | N/A | MIT | BSD-3-Clause |
| 适合场景 | 音乐播放器、音效管理 | 自定义管线 | 音乐创作 / 合成器 | 音频编辑 / 可视化 |

**Why Howler.js**：

1. **移动端音频限制自动处理**：移动浏览器（尤其是 iOS Safari 和 Android WebView）要求音频播放必须在用户手势事件内触发，否则 `AudioContext` 处于 `suspended` 状态。Howler.js 内置 `Howler.autoUnlock` 机制，首次 `touchstart` / `click` 时自动 `resume()` 所有 AudioContext，无需手动编写解锁代码。

2. **自动降级策略**：Howler.js 优先使用 Web Audio API 创建音频节点，若浏览器不支持或 Web Audio 上下文创建失败，自动降级为 HTML5 `<audio>` 元素。在 Capacitor WebView 中，部分厂商 ROM 的 WebView 实现存在 Web Audio 兼容性问题，降级机制保证音频播放不中断。

3. **内置播放列表与精灵图**：`Howl.sprite` 支持将多个音频片段映射到单文件的不同时间偏移，适合音效管理和无缝切歌场景。`Howler.ctx` 暴露原生 `AudioContext` 引用，方便接入自定义音效管线。

4. **事件系统完善**：提供 `load/play/pause/end/seek/fade/volume/rate/loaderror/playerror` 等完整事件回调，无需手动监听 `<audio>` 元素事件或 `AudioContext.statechange`。

5. **稳定成熟**：v2.x 已发布超过 5 年，npm 周下载量 >400K，GitHub Stars >30K，被 Spotify Web Player、Notion 等产品大规模使用。

**架构分工**：

```
Howler.js（播放控制层）           Web Audio API（音效处理层）
────────────────────────        ─────────────────────────
  Howl.play() / .pause()          BiquadFilterNode × 5
  Howl.seek() / .rate()           (均衡器: 60/230/910/3.6k/14k Hz)
  Howl.volume() / .fade()              │
  Howl.sprite（切歌）              GainNode × 3
  Howl.event（事件回调）           (前置增益 / 响度增强 / 主音量)
       │                                │
       └──── Howler.ctx ────────────────┘
            (共用同一个 AudioContext)
```
Howler.js（播放控制层）           Web Audio API（音效处理层）
─────────────────────────       ─────────────────────────
  Howl.play() / .pause()          BiquadFilterNode × 10
  Howl.seek() / .rate()           (均衡器)
  Howl.volume() / .fade()              │
  Howl.sprite（切歌）              GainNode × 3
  Howl.event（事件回调）           (前置增益 / 响度增强 / 主音量)

**完整代码示例**：

```typescript
// src/audio/effectChain.ts
const EQ_FREQUENCIES = [60, 230, 910, 3600, 14000];

class EffectChain {
  private preGain: GainNode;
  private eqFilters: BiquadFilterNode[];
  private bassBoost: BiquadFilterNode;
  private loudnessGain: GainNode;
  private masterGain: GainNode;

  constructor(ctx: AudioContext) {
    this.preGain = ctx.createGain();
    this.eqFilters = EQ_FREQUENCIES.map(() => {
      const filter = ctx.createBiquadFilter();
      filter.type = 'peaking';
      filter.Q.value = 1.0;
      filter.gain.value = 0;
      return filter;
    });
    this.bassBoost = ctx.createBiquadFilter();
    this.bassBoost.type = 'lowshelf';
    this.bassBoost.frequency.value = 80;
    this.bassBoost.gain.value = 0;
    this.loudnessGain = ctx.createGain();
    this.masterGain = ctx.createGain();
  }

  connect(source: AudioNode, destination: AudioNode): void {
    let lastNode: AudioNode = source;
    lastNode.connect(this.preGain);
    lastNode = this.preGain;
    for (const filter of this.eqFilters) {
      lastNode.connect(filter);
      lastNode = filter;
    }
    lastNode.connect(this.bassBoost);
    this.bassBoost.connect(this.loudnessGain);
    this.loudnessGain.connect(this.masterGain);
    this.masterGain.connect(destination);
  }

  setBiquadFilter(index: number, params: { frequency: number; Q: number; gain: number }): void {
    const filter = this.eqFilters[index];
    if (!filter) return;
    filter.frequency.value = params.frequency;
    filter.Q.value = params.Q;
    filter.gain.value = params.gain;
  }

  setMasterVolume(value: number): void {
    this.masterGain.gain.value = value;
  }
}
```

### App 端技术决策

#### 1. 后台播放方案

**选型结论**：自写原生 `Foreground Service` + WebView 双向通信（JS Bridge），Android 侧注册 `MediaSessionService`。

**理由**：Capacitor 社区插件（`@capacitor/background-task`、`@jofr/capacitor-media-session`）仅对 iOS 后台保活有一定效果，Android 端均无法稳定维持长时间 Audio 上下文。Android 自 8.0 起强制前台 Service 必须在 5 秒内调用 `startForeground()` 并绑定持久通知，Capacitor 插件未封装此生命周期，直接用原生更可靠且可深度集成 `MediaSession`。

**完整链路**：

```
Web 层 (React)                      Android 原生层
─────────────────                   ─────────────────
Web Audio API 解码/播放              Foreground Service
  │                                    │
  ├─ JS Bridge: postMessage() ──────►  ├─ startForeground(notification)
  │  {type:'play', title, artist}      │
  │                                    ├─ MediaSession.setMetadata()
  │◄──── JS Bridge: callback ────────  │
  │  {type:'media_event', action}      ├─ MediaSession.Callback
  │                                    │   onPlay/onPause/onSkipToNext...
  │  JS Bridge: postMessage() ──────►  │
  │  {type:'update_metadata', ...}     │
  │                                    └─ 通知栏更新 (封面+标题+按钮)
```

**对比表**：

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| `@capacitor/background-task` 插件 | 纯 TypeScript 调用，无需写原生代码 | 仅适用于短任务（~30秒），不符合长时音频播放场景 | 否决 |
| `@jofr/capacitor-media-session` | 社区封装了 MediaSession，开箱即用 | Android 8.0+ 前台 Service 未绑定，容易被系统杀死；插件维护不活跃，已知 issue 较多 | 否决 |
| **自写原生 Service + WebView 双向通信** | 完全可控，MediaSession + Foreground Service 深度集成，稳定性最高 | 需要编写少量 Java/Kotlin 原生代码，维护成本略高 | **采纳** |

**实现要点**：

```kotlin

    override fun onCreate() {
        super.onCreate()
        val session = MediaSession.Builder(this, player)
            .setCallback(PlaybackCallback())
            .build()
        // 绑定前台通知
        startForeground(NOTIFICATION_ID, buildNotification(session))
    }
}

#### 2. 通知栏 / 锁屏媒体控制

**选型结论**：Android 原生 `MediaSession` + JS Bridge 为主方案，Web `Media Session API` 仅作为 PWA 模式的降级方案。

**理由**：Android 通知栏媒体控制需绑定 `MediaSession.Token` 到 `MediaStyle` 通知，`MediaSessionCompat` 提供完整的封面/标题/按钮定制能力。Web Media Session API 在 WebView 中映射不完整（封面显示不稳定、部分厂商 ROM 不被识别），仅适用于 PWA 浏览器环境。

**对比表**：

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| Web Media Session API 原生映射 | 零原生代码，`navigator.mediaSession.setActionHandler()` 即可 | WebView 内映射不完整；封面图需 HTTPS URL，本地 `file://` 不支持；华为/小米等 ROM 对 WebView 内 MediaSession 支持差异大 | PWA 模式保留，Capacitor 模式不依赖 |
| `@jofr/capacitor-media-session` 插件 | 封装了 Android MediaSession，提供 Capacitor API | 插件已停止维护，不支持 Android 13+ 新权限模型，自定义通知样式受限 | 否决 |
| **Android 原生 MediaSession + JS Bridge** | 完全控制通知栏样式、封面、按钮；兼容所有 Android 厂商 ROM | 需编写原生代码，封面图需通过原生层加载 | **采纳** |

**封面/标题/上下曲渲染方案**：

```
┌─────────────────────────────────────┐
│  ♬ MusicPlayer                     │
│  ┌──────┐                          │
│  │ 封面  │  晴天 - 周杰伦            │
│  │      │  正在播放                  │
│  └──────┘                          │
│  ⏮   ⏯   ⏭    (MediaStyle 按钮)   │
└─────────────────────────────────────┘
```

- 封面：JS 层从音频文件 ID3 标签提取封面 Blob → 通过 Bridge 写到原生临时目录 → `MediaMetadataCompat.Builder.putBitmap()` 设置
- 标题/歌手：JS 层播放器状态变化时通过 Bridge 实时同步 → `MediaMetadataCompat.Builder.putString()`
- 按钮：`PlaybackStateCompat.Builder` 配置 `ACTION_PLAY_PAUSE / ACTION_SKIP_TO_NEXT / ACTION_SKIP_TO_PREVIOUS`
- 进度条：`PlaybackStateCompat.Builder.setBufferedPosition()` + `setPlaybackSpeed()`

#### 3. Android 文件系统访问策略

**选型结论**：分层策略，按 Android 版本走不同通道。

| Android 版本 | 策略 | 技术手段 |
|---|---|---|
| Android 9- (API ≤28) | 传统文件权限 | `READ_EXTERNAL_STORAGE` 运行时权限，直接访问文件系统 |
| Android 10 (API 29) | Scope Storage 过渡 | 清单声明 `requestLegacyExternalStorage=true`，兼容旧行为 |
| Android 11+ (API ≥30) | MediaStore + SAF | `READ_MEDIA_AUDIO` 权限读取媒体库，SAF `ACTION_OPEN_DOCUMENT_TREE` 补充访问自定义目录 |

**对比表**：

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| `@capacitor/filesystem` 插件 | Capacitor 官方插件，API 稳定，支持读/写/列目录 | Android 11+ Scope Storage 下 `Directory.ExternalStorage` 受限，无法遍历用户自定义音乐目录 | 基础层使用，用于读写应用沙盒内文件（封面缓存、封面临时目录） |
| SAF `Intent.ACTION_OPEN_DOCUMENT_TREE` | 用户自主选择目录，兼容 Android 11+，权限持久化 | 需用户手动操作，不能后台静默扫描；需要额外 UI 引导 | 补充方案，作为用户自定义目录入口 |
| **MediaStore API + SAF 组合** | 覆盖系统媒体库 + 用户自定义目录，权限模型清晰，符合 Google Play 政策 | `MediaStore` 扫描全量媒体文件速度较快，但部分厂商 ROM 不会自动索引用户自定义目录 | **采纳** |

**用户授权流程**：

```
App 首次启动
  │
  ├─ 请求 READ_MEDIA_AUDIO (Android 13+)
  │  └─ 弹系统权限对话框 → 用户授权/拒绝
  │
  ├─ 弹出引导页：选择音乐来源
  │  ├─ [扫描系统媒体库]  → MediaStore 自动扫描
  │  └─ [手动选择文件夹]  → SAF ACTION_OPEN_DOCUMENT_TREE
  │       └─ 用户选择后，persistableUriPermission 持久化
  │
  └─ 开始扫描 → 解析 ID3 → 写入 IndexedDB
```

**Capacitor 插件封装**：

```typescript
// src/platform/capacitor.ts
class CapFileSystem implements MusicPlatform {
  async scanFolder(): Promise<AudioFile[]> {
    // Android 11+: 调用原生 MediaStore 查询
    // Android 9-: 使用 @capacitor/filesystem 遍历路径
    const result = await NativeBridge.scanAudioFiles();
    return result;
  }

  async requestPermission(): Promise<boolean> {
    // Android 13+: READ_MEDIA_AUDIO
    // Android 11-12: READ_EXTERNAL_STORAGE
    const { granted } = await Permissions.request({
      name: 'read_media_audio'
    });
    return granted;
  }
}
```

#### 4. App 端 UI 框架决策

**选型结论**：纯手写 Tailwind CSS，不引入 Ionic / Konsta UI 等 Capacitor 配套组件库。

**理由**：项目已用 Tailwind CSS 统一 Web/PWA 端样式，引入 Ionic 会引入大量冗余组件和样式（包体积 +~500KB gzip），且 Ionic 组件的 iOS 风格元素在纯 Android 场景下视觉不协调。Konsta UI 对 Capacitor 的适配较好但组件库不够成熟（GitHub Stars <2k）。纯手写 Tailwind 利用本项目已有的设计系统（Amber 暗色玻璃主题）和 UI 设计稿，一致性最高。

**对比表**：

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| Ionic Framework | 成熟的移动端组件库，内置手势/动画/导航，与 Capacitor 深度集成 | 包体积大（~500KB+ gzip），iOS 风格与纯 Android 场景不协调，需引入 Ionic 路由系统，与已有 React Router 冲突 | 否决 |
| Konsta UI | Tailwind 风格组件库，iOS/Material 双主题，包体积小（~50KB） | 社区不够成熟（GitHub Stars <2k），组件覆盖不全（缺少虚拟滚动、歌词滚动等），长期维护风险 | 否决 |
| **纯手写 Tailwind CSS** | 与 Web 端样式完全一致，零额外依赖，利用已有设计稿和设计系统，包体积最小 | 需手写移动端交互手势（滑动切换、长按菜单等），开发工作量略大 | **采纳** |

**移动端适配方案**：

- Viewport 标签：`<meta name="viewport" content="width=device-width, initial-scale=1.0, viewport-fit=cover">`
- 安全区域适配：Tailwind `safe-area-inset-*` 工具类
- 底部导航栏：CSS `env(safe-area-inset-bottom)` 兼容全面屏手势区域
- 触摸反馈：CSS `-webkit-tap-highlight-color: transparent` + `active:` 伪类
- 手势交互：轻量级 `@use-gesture/react`（~10KB）处理滑动/缩放

#### 5. iOS 端支持计划

**选型结论**：当前阶段（v1-v2）仅支持 Android，架构预留平台抽象层，v3 阶段评估 iOS 支持。

**理由**：
- iOS 端本地文件访问受沙盒限制严重（无法自由访问文件夹，仅能通过 `UIDocumentPickerViewController` 逐文件选择），与"批量扫描文件夹"的核心功能冲突
- iOS App Store 审核对本地音乐播放器有严格的内容版权审查，纯本地播放器上架成功率低
- 当前团队资源集中在 Android 端完善体验，iOS 跨平台 ROI 不高

**跨平台策略**：

    │ (v1-v2) │    │  (PWA)   │    │ (v3 评估)│

**对比表**：

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| v1 同时支持 iOS | 发布即双端覆盖 | iOS 文件访问受限，App Store 上架困难；开发资源分散 | 否决 |

| **v1-v2 仅 Android，架构预留 iOS 扩展点** | 当前资源聚焦 Android，架构不排斥 iOS，后期扩展成本低 | iOS 用户暂时无法使用 | **采纳** |

#### 7. 离线资源打包策略

**选型结论**：APK 内预置 Web 资源（HTML/JS/CSS/图标等静态资源）+ 首次启动不联网直接可用。音频文件不由 APK 携带，由用户自行导入。

**首屏体验与包体积权衡**：

| 资源类型 | 打包策略 | 大小估算 |
|----------|---------|----------|
| React App (JS/CSS) | APK 内 `assets/public/` 预置，Capacitor 默认行为 | ~300KB gzipped |
| PWA 图标 (192/512px) | APK 内 `res/` 打包 | ~30KB |
| 默认均衡器预设 | 代码内常量 | 0KB |
| 音频文件 | 不预置，用户自行导入 | — |
| IndexedDB / Preferences | 首次启动初始化 | — |

**APK 体积预估**：Web 资源 ~350KB + Capacitor 壳 ~3MB + 原生代码 ~500KB ≈ **4MB** (Debug APK)。

**对比表**：

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| 全部资源预置到 APK | 首屏秒开，零网络依赖 | 包体积随资源线性增长（若预置 demo 音频则显著增加） | Web 静态资源部分采纳 |
| 首次启动从 CDN 下载 | APK 体积最小 | 违背"纯本地、不依赖网络"的核心定位 | 否决 |
| Service Worker 协同（首次离线 → SW 缓存 → 二次离线可用） | 兼顾首屏与离线 | SW 在 WebView 中行为不一致（部分厂商 ROM 禁用），且与 Capacitor 本地加载模式冲突 | 否决 |
| **APK 预置 Web 资源 + 用户导入音频** | 首屏零网络依赖，符合本地播放器定位，包体积可控 | 若未来需要预置语音提示或音效，需单独处理 | **采纳** |

**Service Worker 的角色**：

- PWA 模式：SW 负责离线缓存，`vite-plugin-pwa` 自动生成
- Capacitor 模式：SW **不启用**（`vite-plugin-pwa` 构建时跳过），Capacitor 直接从 `assets/public/` 本地加载，SW 缓存在 WebView 中反而产生双重缓存问题

#### 8. Android 端路由策略

**选型结论**：完全共用 React Router v6，不引入任何原生页面。权限引导页通过 React 组件 + 条件渲染实现。

**理由**：Capacitor 的本质是将 Web App 运行在 WebView 中，所有页面应由 React Router 管理以保持代码统一。仅在极端场景（如 Android 13 运行时权限被永久拒绝且无法通过 WebView 唤起设置页）时，通过 JS Bridge 调用原生 `Intent` 跳转系统设置页。

**对比表**：

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| 完全共用 React Router | 代码完全统一，Web/PWA/APK 一套路由，零维护成本 | 权限引导页的"跳转系统设置"按钮需要通过 JS Bridge 调用原生 Intent | **采纳** |
| React Router + 部分原生页面 | 权限引导页原生实现，UI 更原生 | 代码分裂，维护两套 UI；路由跳转复杂（原生 ↔ WebView） | 否决 |
| 完全原生路由（弃用 React Router） | 无 | 放弃 Web/PWA 端，违背双模式架构，完全不可行 | 否决 |

**路由表**：

```typescript
// src/App.tsx
<Routes>
  <Route path="/" element={<HomePage />} />
  <Route path="/player" element={<PlayerPage />} />
  <Route path="/player/:id" element={<PlayerPage />} />
  <Route path="/lyrics" element={<LyricsPage />} />
  <Route path="/equalizer" element={<EqualizerPage />} />
  <Route path="/hidden" element={<HiddenPage />} />
  <Route path="/scan" element={<ScanPage />} />
  <Route path="/search" element={<SearchPage />} />
  <Route path="/permissions" element={<PermissionsGuidePage />} />  {/* 权限引导 */}
</Routes>
```

**权限引导页 (`PermissionsGuidePage.tsx`)**：首次启动检测 `VITE_PLATFORM === 'capacitor'` 且无权限时自动路由到 `/permissions`，引导用户授权。若权限被永久拒绝，提供"前往设置"按钮，通过 JS Bridge 跳转：

    intent.data = Uri.parse("package:$packageName")

#### 10. 性能优化策略

**选型结论**：

| 优化维度 | 选型 | 说明 |
|----------|------|------|
| WebView 硬件加速 | `android:hardwareAccelerated="true"`（默认） | 启用 GPU 渲染，显著提升 CSS 动画帧率 |
| 大歌曲列表虚拟滚动 | `@tanstack/virtual` | 数千首歌曲时仅渲染可视区域 DOM 节点 |

**对比表 — 虚拟滚动**：

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| `react-window` | 老牌虚拟滚动库，API 稳定，社区广泛使用 | 固定行高假设不适合混合内容（歌曲行 vs 分组标题行），动态高度支持有限 | 备选 |

**对比表 — 音频解码线程隔离**：

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| 主线程 `decodeAudioData()` | API 简单 | 大量歌曲扫描时解码阻塞主线程，UI 卡顿 | 否决 |
| **OfflineAudioContext** | 独立解码线程，不阻塞主线程渲染，支持批量处理 | API 稍复杂，需 Promise 管理 | **采纳** |

#### 11. Android SDK 版本策略

**选型结论**：

| SDK 配置 | 取值 | 理由 |
|----------|------|------|
| `minSdkVersion` | **26** (Android 8.0) | Android 8.0 开始对前台 Service 和通知渠道有完整支持；截至 2026 年，Android 8.0 以下设备占比 <2%（Google 官方数据），不值得兼容 |
| `targetSdkVersion` | **34** (Android 14) | 满足 Google Play 最新上架要求 |
| `compileSdkVersion` | **35** (Android 15) | 使用最新 API 编译以获取最新 lint 检查和优化 |

**Capacitor 6 默认 SDK 配置**：

Capacitor 6 创建 Android 项目时默认设置 `minSdkVersion 22`（Android 5.1）、`targetSdkVersion 34`、`compileSdkVersion 34`。本项目基于以下原因将 `minSdkVersion` 提升至 26：

1. **前台 Service 强制要求**：后台播放方案依赖 `MediaSessionService` + 前台通知，Android 8.0 起才提供完整的通知渠道（Notification Channel）体系和前台 Service 生命周期保障。低于 8.0 的设备无法稳定维持后台音频播放。
2. **设备占比极低**：根据 Google 2026 年 5 月发布的 Android 版本分布数据，Android 7.x 及以下设备活跃占比已降至 2% 以下，且以老旧平板/机顶盒为主，非音乐播放器目标设备。
3. **Capacitor 6 官方建议**：Capacitor 官方文档明确指出 `minSdkVersion 22` 仅为最低兼容基线，生产应用应根据实际需求上调。

### 4.1 本地音乐扫描

**扫描流程**：
1. 扫描页显示已有文件夹列表
2. 点击"+"调用系统文件管理器选择新文件夹
3. 选择文件夹后点击"开始扫描"

#### 底部迷你播放栏滑动手势

在底部 `.musicbar` 迷你播放栏上支持滑动手势切换上下曲。

**交互定义**：

| 手势方向 | 操作 | 说明 |
|---------|------|------|
| 左滑（←） | 下一曲 | 手指从右向左滑动 |
| 右滑（→） | 上一曲 | 手指从左向右滑动 |

**触摸手势规范**：

| 事件 | 处理逻辑 |
|------|---------|
| `touchstart` / `mousedown` | 记录起始坐标 `(startX, startY)`，重置位移状态 |
| `touchmove` / `mousemove` | 实时计算水平位移 `deltaX`，迷你封面跟随偏移；垂直方向位移超过水平方向时取消手势识别（防止与页面滚动冲突） |
| `touchend` / `mouseup` | 判断 `abs(deltaX)` 是否 ≥ 阈值，达到则触发切歌，未达到则回弹复位 |

**Swipe 阈值**：水平位移绝对值 ≥ **50px** 时判定为有效滑动手势。低于阈值的滑动视为误触，松手后弹性回弹。

**视觉反馈**：

| 阶段 | 行为 |
|------|------|
| 拖动中 | 迷你封面（`.musicbar-cover`）跟随手指水平偏移，偏移量与 `deltaX` 成正比（带阻尼系数 0.4），曲目信息文字同时淡入淡出过渡 |
| 松手回弹 | 未达阈值时，封面通过 `transition: transform 0.3s ease-out` 弹性回弹至原位 |
| 切歌过渡 | 达到阈值后，封面快速滑出屏幕边缘（200ms），新曲目封面从反向滑入，同时更新歌曲名称与歌手信息 |

**边界行为**：

| 场景 | 行为 |
|------|------|
| 当前为第一首 + 右滑 | 循环至最后一首（列表循环模式）/ 无反应（顺序模式，此时弹性回弹） |
| 当前为最后一首 + 左滑 | 循环至第一首（列表循环模式）/ 无反应（顺序模式，此时弹性回弹） |

**实现要点**：

- 手势识别仅在 `.musicbar` 区域内生效，不干扰歌曲列表和页面滚动
- 横滑与竖滑冲突处理：`abs(deltaX) > abs(deltaY)` 时判定为横向滑动手势，阻止页面默认滚动行为
- PC 端同时绑定 `mousedown` / `mousemove` / `mouseup` 事件，方便桌面浏览器调试演示
- 切歌回调触发后更新 Zustand `playerStore` 中的 `currentTrack`，驱动全局状态同步

### 4.7 歌词格式行业标准

#### 行业格式全景

| 格式 | 全称 | 来源 | 时间精度 | 逐字支持 | 编码 |
|------|------|------|---------|---------|------|
| LRC | LyRiCs | 早期 WinAMP / 千千静听 | 行级（精确到 10ms） | 否 | UTF-8 / GBK |
| **Enhanced LRC** | Enhanced LyRiCs | LRC 扩展（社区规范） | **逐字级（精确到 10ms）** | **是（`<mm:ss.xx>` 逐字标签）** | UTF-8 |
| KRC | KuGou LyRiCs | 酷狗音乐 | 逐字级 | 是（二进制编码 + zlib 压缩） | 二进制 |
| QRC | QQ Music LyRiCs | QQ 音乐 | 逐字级 | 是（XML 结构） | UTF-8 |
| YRC | NetEase LyRiCs | 网易云音乐 | 逐字级 | 是（JSON 结构） | UTF-8 |
| TTML | Timed Text Markup Language | W3C 标准 | 逐字级 | 是（`<span>` 时间区间） | UTF-8 / UTF-16 |
| WebVTT | Web Video Text Tracks | W3C 标准 | 逐字级 | 是（`<c>` 标签 + 时间戳） | UTF-8 |
| ID3v2 SYLT | Synchronized Lyrics/Text | ID3 标准 | 行级 | 否（仅行级时间戳） | 二进制 |
| TXT | 纯文本歌词 | 通用 | 无 | 否 | UTF-8 / GBK |

#### 各格式详解

**1. 标准 LRC（最广泛兼容）**

```
[ti:今后我与自己流浪]
[ar:张碧晨]
[al:哪吒之魔童降世 电影原声带]
[by:王子]
[offset:0]
[00:12.34]我习惯在夜里独自行走
[00:16.78]穿过城市的每个路口
[00:21.45]我习惯在雨中不撑伞
[00:25.90]任凭雨水打湿我的头
```

- 元数据标签：`[ti:]` 标题、`[ar:]` 歌手、`[al:]` 专辑、`[by:]` 作者、`[offset:]` 全局偏移（毫秒）
- 时间戳格式：`[mm:ss.xx]` 或 `[mm:ss.xxx]`，精度 10ms
- 仅行级粒度，无逐字信息

**2. Enhanced LRC（逐字卡拉OK行业事实标准）**

```
[ti:今后我与自己流浪]
[ar:张碧晨]
[00:12.34]我<00:12.34>习<00:12.68>惯<00:13.02>在<00:13.36>夜<00:13.70>里<00:14.04>独<00:14.38>自<00:14.72>行<00:15.06>走
[00:16.78]穿<00:16.78>过<00:17.12>城<00:17.46>市<00:17.80>的<00:18.14>每<00:18.48>个<00:18.82>路<00:19.16>口
```

- 每个字后用 `<mm:ss.xx>` 标记该字的起始时间
- 最后一个字无时间标签，持续到行结束或下一行开始
- 兼容标准 LRC 解析器（逐字标签可被忽略，行级时间戳仍有效）
- 无官方 RFC，但被 QQ 音乐 / 网易云 / 酷狗 / 酷我 等主流播放器广泛导入导出

**3. KRC（酷狗私有格式）**

- 二进制格式，zlib 压缩存储
- 逐字时间信息编码在压缩数据中
- 不支持纯文本编辑，需专用工具解码
- 酷狗音乐独占使用，几乎无第三方工具支持

**4. QRC（QQ 音乐私有格式）**

```xml
<?xml version="1.0" encoding="UTF-8"?>
<QrcInfos>
  <QrcHeadInfo SaveTime="200" Version="100"/>
  <LyricInfo>
    <Lyric_1 LyricType="1" LyricContent="
      [00:12.34]我(12134,340)习(12474,340)惯(12814,340)在(13154,340)夜(13494,340)
里(13834,340)独(14174,340)自(14514,340)行(14854,340)走(15194,340)
    "/>
  </LyricInfo>
</QrcInfos>
```

- XML 结构，每字格式 `字(起始毫秒,持续毫秒)`
- 精确到毫秒，QQ 音乐独占
- 体积较大（XML 序列化冗余），不适合大量存储

**5. YRC（网易云音乐私有格式）**

```json
{
  "t": 0,
  "c": [
    {"t": 12340, "c": [{"t": 0, "c": "我"}, {"t": 340, "c": "习"}, ...]}
  ]
}
```

- JSON 结构，使用相对时间（每个字相对于行起点）
- 网易云音乐独占，第三方支持极少

#### 选型决策

| 维度 | Enhanced LRC | KRC | QRC | YRC | TTML/WebVTT |
|------|-------------|-----|-----|-----|-------------|
| 文本可编辑性 | ✅ 纯文本 | ❌ 二进制 | ⚠️ XML | ⚠️ JSON | ⚠️ XML |
| 逐字精度 | ✅ 10ms | ✅ 毫秒 | ✅ 毫秒 | ✅ 毫秒 | ✅ 毫秒 |
| 兼容标准 LRC | ✅ 向下兼容 | ❌ | ❌ | ❌ | ❌ |
| 主流播放器支持 | ✅ 广泛导出 | ❌ 仅酷狗 | ❌ 仅QQ | ❌ 仅网易云 | ⚠️ 部分 |
| 文件体积 | ✅ 极小 | ✅ 极小（压缩） | ❌ 大 | ⚠️ 中等 | ❌ 大 |
| 社区工具链 | ✅ 成熟 | ❌ 无 | ❌ 极少 | ❌ 极少 | ⚠️ 有 |
| 学习成本 | ✅ 低 | ❌ 高 | ⚠️ 中 | ⚠️ 中 | ⚠️ 中 |

**最终选型：Enhanced LRC**

理由：
1. **纯文本可手工编辑**，用户可自行制作或从网上下载逐字歌词
2. **向下兼容标准 LRC**，无逐字信息时自动按等分策略生成逐字时间
3. **主流播放器广泛导出**，QQ 音乐/网易云/酷狗均可导出为 Enhanced LRC
4. **体积最小**，100 行歌词约 3-5KB
5. **社区工具链成熟**，有 `lrc-kit`、`lrc-parser` 等现成 npm 包

**降级策略**：
- 优先读取音频同目录下的 `.lrc` 文件（Enhanced LRC）
- 若无 `.lrc`，回退读取嵌入歌词（ID3v2 USLT 帧，仅行级）
- 行级歌词无逐字信息时，按该行起始时间 + 总时长等分每字
- 若无任何歌词，显示"纯音乐，请欣赏"

#### 元数据字段映射

| LRC 标签 | 映射字段 | 说明 |
|---------|---------|------|
| `[ti:]` | song.title | 歌曲标题 |
| `[ar:]` | song.artist | 歌手名 |
| `[al:]` | song.album | 专辑名 |
| `[by:]` | lyric.author | 歌词作者 |
| `[offset:]` | lyric.offset | 全局时间偏移（毫秒），+延迟 -提前 |
| `[la:]` | lyric.language | 歌词语言（非标准，部分社区扩展） |

### 4.8 逐字歌词

**数据来源**：与音频文件同目录下的同名 `.lrc` 文件，优先读取嵌入歌词（ID3v2 USLT 帧）。

**LRC 解析器 (`src/lyrics/parser.ts`)**：

```typescript
// 解析标准 LRC 格式，输出按时间排序的词条数组
interface LyricWord {
  text: string       // 单字文本
  startMs: number    // 开始时间(毫秒)
  durationMs: number // 持续时间(毫秒)
}

interface LyricLine {
  text: string           // 整行文本
  startMs: number        // 行开始时间
  words: LyricWord[]     // 逐字切分
}

// 支持标准行级 LRC ([mm:ss.xx]歌词) +
// 逐字 LRC ([mm:ss.xx]<mm:ss.xx>字) 变体
// 无逐字标记时自动按总时长等分每行
parseLRC(raw: string, durationMs: number): LyricLine[]
```

**逐字高亮算法（三色卡拉OK，参考 QQ音乐/网易云/Spotify）**：

> 核心原则：纯文字变色，禁止字符矩形背景块（`background` / `padding` / `border-radius`）。

| 字状态 | CSS class | 视觉效果 | 样式要点 |
|--------|-----------|---------|---------|
| 未唱字（默认） | — | 白色文字，无特效 | `color: oklch(0.93 0.005 270)` |
| 已唱字 | `.sung` | accent 主题色，opacity 0.5 | 不发光，不缩放 |
| 当前字 | `.active` | accent + 发光 + scale(1.1) + 加粗 | `text-shadow: 0 0 12px var(--accent-glow)`，单字过渡 300ms `cubic-bezier(0.34, 1.56, 0.64, 1)` |

#### 卡拉OK显示规范（参考 QQ音乐 / 网易云音乐 / Spotify）

##### 色板定义

| 角色 | 颜色值 | 用途 |
|------|--------|------|
| 未唱字 | `oklch(0.93 0.005 270)` ≈ `#ECEDF0` | 默认字符色，无特效 |
| 已唱字 | `oklch(0.72 0.16 82)` opacity 0.5 | 琥珀金半透明，不发光 |
| 当前字 | `oklch(0.72 0.16 82)` + `text-shadow: 0 0 12px oklch(0.62 0.14 80)` | 琥珀金发光 + scale(1.1) + font-weight 700 |
| 过往行 | `opacity: 0.28` | 行整体淡出 |
| 当前行 | `opacity: 1`，`font-size: 1.15rem` | 视觉焦点行 |
| 未来行 | `opacity: 0.42` | 行半透明 |

##### 逐字过渡时序

```
行N-2 (past)     opacity 0.28  ────────────────────────────────
行N-1 (past)     opacity 0.28  ────────────────────────────────
行N   (current)  字1→字2→字3→字4→字5→字6→ ... →字10
                  ↑已唱↑已唱↑active↑未唱↑未唱↑未唱    ↑未唱
                        (amber glow)
行N+1 (future)   opacity 0.42  ────────────────────────────────
行N+2 (future)   opacity 0.42  ────────────────────────────────
```

- 每次 `activeWordIdx` 变化时，前一个字从 `.active` → `.sung`（300ms transition）
- 新当前字添加 `.active`（即时，无入场动画延迟）
- glow 效果范围：当前字为中心，12px blur radius
- 禁止字符矩形背景块（`background` / `padding` / `border-radius`），纯文字变色

##### 行切换动画

```
activeLineIdx: N → N+1
─────────────────────────
行N:   .current → .past  (opacity: 1 → 0.28, transition 400ms ease-out)
行N+1: .future → .current (opacity: 0.42 → 1, font-size → 1.15rem, transition 400ms ease-out)
行N+2: .future → .future (无变化，仍在可视区)
```

- 行切换触发条件：当前行最后一个字变为 `.sung`，且下一行第一个字的时间戳到达
- transition 使用 `cubic-bezier(0.4, 0, 0.2, 1)` (Material Design standard easing)
- 禁止在行切换期间触发 `scrollTo`，等 opacity transition 完成后再调整滚动位置

##### 自动滚动策略

| 规则 | 值 |
|------|-----|
| 触发时机 | `activeLineIdx` 变化时触发一次（不是每帧） |
| 滚动方式 | `scrollTop` 直接赋值，非 `scrollTo({ behavior: 'smooth' })` |
| 目标位置 | 当前行垂直居中于视口（`line.offsetTop - viewportHeight/2 + line.height/2`） |
| 理由 | RAF 60fps 下每帧调 smooth scroll 会堆积动画队列，导致滚动完全失效 |

##### 虚拟播放时钟

```
时间源: RAF timestamp 差值累加 → virtualClock
       (禁止 Date.now() 或 performance.now() 直接读取)

每一帧:
  virtualClock += deltaMs
  activeLineIdx = binarySearch(lines, virtualClock)
  activeWordIdx = binarySearch(currentLine.words, virtualClock)
```

- 用 RAF `timestamp` 差值累加而非直接读系统时间，避免 tab 切换后时间跳跃
- binary search 复杂度 O(log n)，保证 60fps 下性能

##### 性能策略

| 对象 | 策略 |
|------|------|
| 当前行字 | 启用 `will-change: transform, opacity` |
| 非当前行 | `content-visibility: auto`（跳过渲染不可见行） |
| 歌词列表 | 虚拟滚动：仅渲染当前行 ± 10 行，其余用 `padding-top/bottom` 占位 |
| DOM 更新 | 仅更新 `className`，不替换 DOM 节点 |

**行级三层渐变**：

| 行状态 | CSS class | 样式 | 说明 |
|--------|-----------|------|------|
| 过往行 | `.past` | `opacity: 0.28` | 淡出，不干扰视线 |
| 当前行 | `.current` | `opacity: 1`，`font-size: 1.15rem` | 视觉焦点，左侧进度细线从下到上填充 |
| 未来行 | `.future` | `opacity: 0.42` | 半透明，暗示即将到来 |

- 进度细线 `.lyric-line-progress`：仅当前行显示，高度 0%~80%，反映当前行播放进度。

**动画引擎**：
- 驱动：`requestAnimationFrame`（RAF），时间源为 RAF `timestamp` 差值累加虚拟播放时钟
- 严禁 `setTimeout` / `setInterval`，严禁每帧调用 `scrollTo({ behavior: 'smooth' })`

**自动滚动规则**：
- 仅在 `activeLineIdx` 变化时触发一次滚动
- 使用 `scrollTop` 直接赋值居中当前行，非 `scrollTo({ behavior: 'smooth' })`
- 原因：RAF 60fps 下每帧调 smooth scroll 会堆积动画队列导致滚动完全失效

### 4.9 桌面歌词浮窗（仅 PWA / Desktop 模式）

**浮窗三色同步**：浮窗内歌词与主歌词页共享同一逐字高亮引擎，显示当前行（`.current`）+ 下一行（`.future`），同主歌词的三色逐字逻辑（已唱字 sung、当前字 active+glow、未唱字白色）。

### 4.10 睡眠模式

**需求描述**：用户可在播放页一键设置定时关闭，到时间后自动暂停播放。解决睡前听歌忘记关的痛点。

**入口**：播放详情页快捷操作行中的时钟图标 🕐。

**交互流程**：
1. 点击时钟图标 → 从底部滑出睡眠模式面板
2. 面板内选择倒计时时长：
   - 不开启（默认）
   - 15 分钟后
   - 30 分钟后
   - 一小时后
   - 自定义（弹出时间选择器）
3. 附加开关："播放完当前歌曲再关闭"（默认开启）
   - 开启时：即使倒计时到，也等当前歌曲播放完毕后再暂停
   - 关闭时：倒计时一到立即暂停
4. 选中后自动开始倒计时，面板关闭
5. 倒计时剩余 1 分钟时在顶部通知栏显示提醒
6. 倒计时结束：自动暂停播放、关闭 Foreground Service、释放音频焦点

**状态管理（Zustand store）**：

```typescript
interface SleepTimerState {
  enabled: boolean           // 是否已开启睡眠模式
  duration: number | null    // 剩余秒数
  finishAfterCurrent: boolean // 播放完当前歌曲再关闭
  startTimer(duration: number, finishAfterCurrent: boolean): void
  cancelTimer(): void
  getRemaining(): number | null
}
```

**技术实现**：
- 倒计时：`setInterval` 每秒递减，剩余 ≤60 秒时触发通知提醒
- 播放完当前歌：监听 Howler.js 的 `end` 事件，结合 `finishAfterCurrent` 标志位决定是否暂停
- 停止播放：`howler.stop()` + `NativeBridge.stopForegroundService()`
- 倒计时数据持久化到 `@capacitor/preferences`，App 重启后恢复

**UI 设计（列表式选择面板）**：
- 底部滑出面板，深色玻璃态卡片（琥珀金主题适配）
- 标题："睡眠模式"
- 选项列表行，选中项右侧显示 ✓（琥珀色）
- 选项：不开启 / 15分钟后 / 30分钟后 / 一小时后 / 自定义
- 附加开关："播放完当前歌曲再关闭"（橙色滑块）
- 点击面板外遮罩区域可关闭

### 4.11 搜索功能

**需求描述**：用户可快速检索本地音乐库中的歌曲，支持按歌曲名、歌手名、专辑名搜索，实时过滤本地数据，无需网络。

**搜索入口**：首页顶部导航栏右侧搜索图标（🔍），点击后展开搜索面板（覆盖层或新页面）。

**搜索维度**：

| 维度 | 匹配字段 | 示例 |
|------|---------|------|
| 歌曲名 | `song.title` | 输入"晴天" → 匹配"晴天 - 周杰伦" |
| 歌手名 | `song.artist` | 输入"周杰伦" → 匹配所有周杰伦的歌曲 |
| 专辑名 | `song.album` | 输入"叶惠美" → 匹配专辑内的歌曲 |

**搜索模式**：纯本地实时过滤。所有歌曲元数据已存储在 IndexedDB / Zustand store 中，搜索时直接从内存/本地数据库读取并过滤，不发起任何网络请求。

**搜索结果 UI**：

- **搜索面板布局**：顶部搜索输入框（带返回/关闭按钮），下方实时过滤结果列表
- **列表项**：复用 `SongItem` 组件，展示歌曲名、歌手名、专辑名、音质标签
- **关键词高亮**：匹配到的关键词在歌曲名/歌手名/专辑名中以主题色（`var(--accent)`）高亮显示
- **排序**：按匹配优先级排序 — 歌曲名精确匹配 > 歌曲名部分匹配 > 歌手名匹配 > 专辑名匹配

**空状态 / 无结果提示**：

- 搜索无匹配结果：居中显示"未找到相关歌曲"提示文字 + 搜索图标，提示用户尝试其他关键词
- 搜索输入为空：显示全部歌曲列表

**去抖策略**：

- 输入事件触发 300ms debounce 后再执行过滤
- 使用 `setTimeout` + `clearTimeout` 实现，避免每次按键都触发全量过滤
- 去抖期间输入框实时显示输入内容（无延迟），仅过滤逻辑延迟执行

**状态管理**：

```typescript
interface SearchState {
  query: string                    // 当前搜索关键词
  results: Song[]                  // 过滤后的结果列表
  isOpen: boolean                  // 搜索面板是否打开
  setQuery(q: string): void
  close(): void
}
```

**技术实现**：

- 搜索过滤函数：`songs.filter(s => s.title.includes(query) || s.artist.includes(query) || s.album.includes(query))`
- 关键词高亮：将匹配文本按关键词拆分为片段，匹配部分包裹高亮标签
- debounce 使用 `setTimeout` / `clearTimeout` 实现 300ms 延时过滤
- 搜索结果列表与歌曲列表共享虚拟滚动组件（`@tanstack/virtual`）

### 扫描页文件选择器

- 原型中通过模拟按钮添加文件夹
- 实际实现需调用系统文件管理器（Android: `Intent.ACTION_OPEN_DOCUMENT_TREE`，iOS: `UIDocumentPickerViewController`）

### 滚动行为

- `body { overflow: hidden }` — 页面不滚动
- `.workspace { overflow-y: auto; height: 100vh }` — 工作区可滚动
- `.page-content { overflow-y: auto; scrollbar-width: none }` — 手机内部可滚动，滚动条隐藏
